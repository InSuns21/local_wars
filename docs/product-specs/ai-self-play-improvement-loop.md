# ai-self-play-improvement-loop

AI エージェントが AI 同士の自己対戦結果を読み取り、改善提案、ソース改修、検証、再対戦までを反復するための正式仕様です。バランス調整、退行検知、思考ルーチン改善を継続的に進めるための運用基準を定義します。

最終更新: 2026-03-22
状態: Phase 3 実装済み（nightmare向け詳細指標・改善提案Markdown自動生成・限定autotuneまで対応）

## 対象スコープ
- AI vs AI の自動対戦実行
- 対戦ログと集計レポートの生成
- AI エージェントによるレポート読解
- レポートに基づく改善提案
- 限定範囲のソース改修
- テストと再対戦による再評価

## ゴール
- 特定 AI プロファイルや戦術の極端な偏りを早期に発見する
- 勝率だけでなく、ターン数、被害、占領、補給、視界獲得など複数指標で AI 品質を評価する
- 改修前後で挙動差分を比較し、退行を検知できるようにする
- 人手レビューの負荷を下げつつ、無秩序な自動最適化を防ぐ

## 基本方針
- 自己対戦は、実プレイヤー向け UI ではなく検証用の評価ループとして扱う
- AI エージェントは、毎回のレポートを根拠に仮説を立てて改修する
- 1 回の対戦結果ではなく、一定試行数の集計結果を評価対象にする
- 勝率のみで最適化せず、ゲーム体験を損なう一方的戦術の過学習を避ける
- 直接の自動改修は限定スコープで始め、重要変更は人手レビュー前提とする

## 標準ループ
1. 対象 AI プロファイル、マップ、設定、試行回数を決める
2. AI vs AI の自己対戦をまとめて実行する
3. 対戦結果を JSON と Markdown の両方で保存する
4. AI エージェントがレポートを読み、偏りや異常を抽出する
5. 改修候補を列挙し、対象ファイルと意図を明示する
6. 限定範囲のコード変更を行う
7. `typecheck` と関連テストを実行する
8. 同一条件で再度 AI vs AI を実行する
9. 改修前後の差分レポートを作成する

## 評価指標
- 勝率
- 平均ターン数
- HQ 制圧率
- 施設占領率
- 資金消費と残高推移
- ユニット生存率
- 補給切れ発生回数
- 可視敵発見数
- 対ドローン、対空、対潜など特定兵科への偏り

## `nightmare` 調整用の評価指標セット
- 表示上の難易度名は `めちゃつよ` とする
- `nightmare` は純粋な強さを優先するため、通常難易度より評価指標を厳しく持つ
- 単一の勝率だけで調整せず、以下の指標を組み合わせて採用可否を決める
- 必須指標
  - 総合勝率
  - 先手時勝率と後手時勝率
  - マップ別勝率
  - Fog of War 有無別勝率
- 攻勢指標
  - 平均 HQ 到達ターン
  - 敵施設占領率
  - 高額敵ユニット撃破効率
  - 勝利確定局面からの取りこぼし率
- 守勢指標
  - 自軍 HQ 被脅威回数
  - 自軍施設喪失率
  - 高額自軍ユニットの不利交換率
  - 補給切れによる行動不能率
- 運用品質指標
  - 兵科構成の偏り
  - 索敵役の生存率
  - 補給役の稼働率
  - ドローン、対空、対潜への対応成功率
- 安定性指標
  - 連戦時の勝率ぶれ幅
  - 特定プロファイルに対する極端な相性差
  - 改修前後での回帰発生数
- `nightmare` 採用時は、勝率が上がっても守勢指標や安定性指標が大きく悪化する変更を避ける

## レポート要件
- 各試合の勝者、ターン数、主要イベントを残す
- 集計レポートには試行条件を必ず明記する
- AI プロファイルごとの相性表を持てる構造にする
- 改修前後の比較では、指標差分を同一フォーマットで出力する
- 生成物は機械処理しやすい JSON と、人間確認しやすい Markdown を併置する

## AI エージェントの責務
- レポートから問題仮説を抽出する
- 仮説に対応する最小限の改修案を作る
- 変更対象は `aiTurn.ts`、関連型、テストなど AI 思考と検証に関係する範囲を優先する
- 変更理由と期待効果を明記する
- 改修後は必ず再検証し、改善しなかった場合はその旨を記録する

## 自動改修の制約
- 1 ループで同時に変える論点は少数に絞る
- 勝率だけを理由に UI、保存、表示系の無関係なコードを変更しない
- 指標悪化やテスト失敗が出た場合は、そのループの改修を採用しない
- ノイズの大きい少数試行だけで重い調整を確定しない
- AI の内部推定や見えない情報を、実プレイヤー向け仕様へ漏らさない

## 導入段階
### Phase 1: 自動対戦と集計
- 固定条件で AI vs AI を複数回実行する
- JSON レポートを生成する
- 手動レビュー用の Markdown 要約を生成する
- 実装済み: `npm run selfplay` で headless 実行し、`docs/generated/self-play/` 配下へ JSON / Markdown を出力できる
- 実装済み: baseline JSON を指定すると差分比較 JSON / Markdown を追加出力できる

### Phase 2: 改修提案の自動化
- AI エージェントがレポートから改善候補を列挙する
- まだ自動コード変更は行わず、提案書までを自動化する

### Phase 3: 限定的な自動改修
- 変更対象を AI 思考ルーチンとテストに限定して自動改修する
- 改修後に同条件の再対戦を実行し、差分レポートを出す
- 実装済み: `npm run selfplay:autotune` で、`src/core/engine/aiNightmareTuning.ts` だけを限定更新し、同条件の before / after / compare レポートを出力できる
- 実装済み: autotune は `nightmare` 参加者のみを対象にし、`killBias / safetyBias / hqPressureBias / antiAirBias / droneBias / navalBias / supplyBias / scoutBias` の調整に限定する

### Phase 4: 継続運用
- 定期実行や PR 前検証に組み込む
- 退行検知と長期バランス監視に使う

## 保存場所の方針
- 実行計画は `docs/exec-plans/` に置く
- 正式仕様は `docs/product-specs/` に置く
- 自動生成レポートは `docs/generated/` または将来の専用ディレクトリに置く
- 長期の改善履歴は比較しやすい命名規則で保存する

## 受け入れ条件
- 同一条件の自己対戦を複数回まとめて実行できる
- 集計結果が JSON と Markdown で出力される
- 改修前後の差分比較ができる
- AI エージェントの提案が、レポートの数値や傾向と結びついている
- 自動改修を有効にする場合でも、テスト失敗時は採用されない

## 残課題
- 試行回数の標準値と打ち切り条件の定義
- マップ相性と AI プロファイル相性の分離評価
- ランダム性が高い条件での統計的な扱い
- 自動改修をどの範囲まで許可するかの運用境界
